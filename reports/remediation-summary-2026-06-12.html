Stage Outcomes

1. Runtime and accessibility

2. Persistence consistency

Session completion, manual item logging, trim, and delete flows now coordinate item totals server-side and validate records before writing.

Fixed

3. Ordering and bulk writes

Normal edits cannot overwrite manual order; reorder, statistics reset, and factory reset use narrow server-side atomic routes.

Fixed

4. Live session lifecycle

Recorder/metronome startup is cancellation-safe, recorder stop handles inactive/error states, and pending BPM writes settle before completion records.

Fixed

5. Invalid templates

Smart block builders now return explicit invalid plans, and live pages show a configuration error instead of hanging on startup.

Fixed

6. Product UX

Start-session controls are higher on item pages, no-BPM modes avoid misleading ladder stats, Free Play is discoverable, and settings template drafts are explicit.

Fixed

7. Maintainability

Main Architecture Changes

Area What changed Why it matters
Repository/API writes Completion and rollback flows use atomic server routes and a completion ledger. Session history and item totals are less likely to drift after retries, stale clients, or interrupted writes.
Session runtime Startup cancellation, recorder cleanup, pause guards, and shared artifact builders were added. Live sessions now have fewer race conditions and less duplicated finalization logic.
Template validation Song/exercise block builders expose block-plan errors instead of returning empty runnable bodies. Bad saved templates become fixable configuration states, not a broken session screen.
Future recordings Latest recording metadata now includes itemKind, itemId, and sessionId. Persistent recordings can be added later without ambiguous song/exercise/free-play ownership.
Future PWA/offline Added docs/offline-write-queue-strategy.md as the required queue design before service-worker caching. Offline support now has a data-consistency gate instead of being a cache-only feature.
