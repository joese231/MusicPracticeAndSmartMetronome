What Is Solid

These parts should be preserved rather than replaced.

Repository seam

Stores depend on src/lib/db/repository.ts, which keeps storage strategy swappable. This is the right starting point for cloud sync or an offline repository.

Pure session math

Tempo stepping, duration allocation, block building, and driver state are mostly pure functions with meaningful tests. This lowers regression risk.

Metronome engineering

Shared AudioContext, prerendered click buffers, drift recovery, and diagnostics show that the audio foundation has already handled real browser timing problems.

Priority Fixes

P0 items are the ones I would fix before adding new major features.

P0

Paused sessions can still advance

The hook exposes raw advance with no paused guard, keyboard Space calls it directly, and the pause overlay uses pointer-events-none. A paused session can move to the next block and restart metronome behavior while visually paused.

Runtime correctness src/lib/session/usePracticeSession.ts:158 src/components/session/SessionShell.tsx:251
P0

Session completion is not crash-atomic across files

/api/sessions/complete appends a session, then separately updates song or exercise totals. If the process fails between writes, retry sees a duplicate session and can skip the missing total update.

Persistence consistency src/app/api/sessions/complete/route.ts:123
P0

Start lifecycle can run audio before the visible timer

The metronome can start before microphone permission resolves, while the session timer and UI only move to started afterward. If permission stalls or the component unmounts, lifecycle cleanup is not fully cancellation-safe.

Reorder can be overwritten by stale item edits

Reorder routes mutate only sortIndex without updating updatedAt, while normal item PATCH payloads can include sortIndex. A stale edit can restore an old order after drag/drop.

Important Next

These are not foundational failures, but they will keep producing bugs if ignored.

Invalid smart templates need explicit build errors

buildBlocks and buildExerciseBlocks can return only warm-up or no body blocks when allocation is invalid. Return a discriminated result like { status: "invalidTemplate" } and show a configuration screen instead of letting auto-start fail silently.

Manual log, trim, and delete are split transactions

Manual item logs append sessions without updating item totals. Trim/delete update the session first, then item totals in a second request. Move these through unified server endpoints that apply the session change and item-time delta together.

Recording stop needs inactive/error handling

SessionRecorder.stop() can hang if the recorder is already inactive and does not reject on recorder errors. Make stop/cancel state transitions explicit before persistent recordings are added.

Session schema validation lags item validation

Song and exercise routes validate carefully, but /api/sessions accepts under-checked records. Add validateSessionRecord for live, manual, free-play, and import paths.

Detail pages put history before starting

On song and exercise detail pages, the primary practice action sits below history and charts. Move current BPM/mode and Start first, then show history below.

Mode-aware copy is missing in places

Timed or no-metronome songs can still show ladder-derived BPM language. Hide target, overspeed, slow reference, and warm-up copy when those concepts do not apply.

Maintainability

The app is still workable, but several modules are becoming change magnets.

Bottom Line

The architecture should be evolved, not replaced.

The core design is defensible for a local-first v1 app. The most important correction is to stop treating multi-file JSON writes as transactional. The next most important correction is to make the live-session hook own only safe, guarded state transitions. After that, the product design improvements are straightforward: put practice-start actions first, make copy mode-aware, and split large UI modules before adding cloud sync, PWA, or persistent recordings.
