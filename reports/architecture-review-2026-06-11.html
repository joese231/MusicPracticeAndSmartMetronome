Architecture Audit

Guitar Song Practice Metronome

Verdict What is solid Critical risks Session model Persistence Frontend and UX Roadmap Subagent split
Reviewed June 11, 2026

The app is not flawed from the ground up. The core idea is sound, but two persistence bugs need priority.

The best parts are the pure session logic, repository boundary, metronome scheduler design, and focused dark UI. The biggest risks are not conceptual. They are consistency issues: file-backed read-modify-write races, statistics totals drifting from session records, duplicated song/exercise session code, and a few UI surfaces that are too desktop-heavy.

What Is Solid

Domain Core

Tempo math, duration allocation, block building, and the driver state machine are mostly pure functions with focused tests.

  • src/lib/session is the right place for practice rules.
  • Derived tempos are recomputed instead of stored.
  • The driver has a small, understandable state shape.

Persistence Boundary

The Repository interface is a good seam. It keeps UI code from depending directly on JSON files or future cloud storage.

  • JSON write-then-rename is the right primitive.
  • Lazy normalization keeps legacy rows readable.
  • Session append IDs are retry-friendly.

Product Design

The product is organized around real practice workflows instead of marketing pages or generic dashboards.

  • Live session screen correctly prioritizes BPM and timer.
  • Stats and method pages reinforce the training model.
  • The visual language is quiet and appropriate for repeated use.

Critical Risks

P0

File read-modify-write can lose data

API routes read JSON, mutate a whole array, then call an atomic write. The lock protects the write step, but not the read plus mutation. Two near-simultaneous appends can both read the old file and the later write can drop the earlier session.

Evidence: src/app/api/sessions/route.ts:29, src/app/api/sessions/[id]/route.ts:33, src/lib/db/fileStore.ts:42

Fix: add a server-side updateJsonAtomic(name, fallback, mutator) helper where read, validation, mutation, and write happen under one per-file lock.

P0

Reset statistics can be undone by the practice-time mirror

resetAllStatistics() clears song totals and sessions, but does not clear data/practice-time.json. On the next song load, useSongsStore can read the mirror and write the old larger total back to the song.

Evidence: src/lib/db/fileRepository.ts:261, src/lib/store/useSongsStore.ts:60, src/app/api/practice-time/route.ts

Fix: either delete the mirror and derive totals from session records, or clear it in reset/factory reset and document exactly which source is authoritative.

Session Model

Assessment:

The session core is well designed. The main issue is that the runtime shell has outgrown the page components, and songs/exercises now duplicate behavior in separate places.

Best next abstraction: a shared PracticeItemRuntime or TempoSubject plus one reusable session shell. Songs and exercises should differ by item policy, not by copy-pasted pages.
P1

Song and exercise session code is duplicated and drifting

The song and exercise pages both own metronome lifecycle, recording, keyboard shortcuts, routing, pause logic, block transitions, session records, and BPM edits. That makes every future session behavior change twice as risky.

Evidence: src/app/session/[id]/page.tsx:86, src/app/exercise-session/[id]/page.tsx:64

Fix: extract usePracticeSession for lifecycle/state and SessionShell for layout. Keep item-specific code in adapters: label, next route, promotion target, and record shape.

P1

Template progression semantics are broader than runtime behavior

ProgressionRule allows trouble, but runtime support is effectively song trouble-spot expansion only. Non-trouble rows and exercise rows ignore that option.

Evidence: src/types/song.ts:53, src/lib/session/blocks.ts:272, src/lib/session/exerciseBlocks.ts:134

Fix: validate that trouble only appears on song trouble rows, or remove it from the generic recipe model and make it a song-specific expansion rule.

P1

Duration contract is inconsistent

Validation rejects fixed-only templates that underfill the selected session length, but allocation accepts them. This can make "planned minutes" disagree with actual scheduled block time.

Evidence: src/lib/session/duration.ts:102, src/lib/session/duration.ts:139

Fix: choose one rule. If selected minutes means planned body length, builders and validators should both reject underfill. If it means allocation cap, rename the UI language.

P1

Invalid BPM relies too much on form validation

Nullable song BPM can become 0 for tempo functions, and scheduler entry points do not enforce a valid playable BPM.

Evidence: src/lib/session/tempo.ts:13, src/lib/metronome/scheduler.ts:299

Fix: add runtime guards before scheduler start/setBpm and return explicit no-metronome behavior for unplayable tempos.

Persistence

P1

Session completion is not transactional

Ending a session updates item totals and appends the session record independently. A failure in either path can leave stats and history disagreeing.

Evidence: src/app/session/[id]/page.tsx:240, src/app/session/[id]/page.tsx:263, src/app/exercise-session/[id]/page.tsx:220

Fix: add a repository operation such as completeSession(record, itemDelta) that appends the record and updates the item total in one server mutation.

P1

Full-row upserts can overwrite newer data

Item edits replace the full saved row from the client snapshot. A stale edit can overwrite practice totals, sort order, or another tab's template changes.

Evidence: src/lib/db/fileRepository.ts:138, src/lib/db/fileRepository.ts:183

Fix: use patch APIs for specific mutations, or add revision checks to reject stale writes.

P1

API boundaries need stronger validation

Songs, exercises, and settings routes accept broad objects or arrays. A malformed client write can poison JSON and break later app loads.

Evidence: src/app/api/songs/route.ts:15, src/app/api/exercises/route.ts:15, src/app/api/settings/route.ts:17

Fix: add strict hand validators or a schema parser before writing any JSON file.

Frontend and UX

P1

Settings has confirmed mobile overflow

At a 390px viewport, the settings page overflowed horizontally. The metronome volume row combines a fixed-width range input, percentage label, and preview button inside a non-wrapping row.

Evidence: src/app/settings/page.tsx:272, src/app/settings/page.tsx:411

Fix: make settings rows stack or wrap on small screens, and use w-full max-w-* instead of fixed range widths.

P1

Live session layout is too desktop-first

The live session uses h-screen overflow-hidden with a large non-wrapping toolbar. On smaller screens, controls or instructions can be clipped.

Evidence: src/app/session/[id]/page.tsx:955, src/app/exercise-session/[id]/page.tsx:840

Fix: use min-h-dvh, allow vertical scrolling where needed, and move secondary controls into a compact menu or second row.

P1

Template editor exposes too much at once

Each block row shows every advanced option inline. The power is useful, but basic edits like toggling blocks or changing minutes are buried in a dense form.

Evidence: src/components/session/BlockTemplateEditor.tsx:195

Fix: collapse blocks by default. Show name, enabled state, duration, tempo, and promotion in the row; expand one block for advanced editing.

P2

Reorder and segmented controls need accessibility polish

Drag reorder is mouse-centric, and several segmented controls are visually clear but incomplete as keyboard-accessible radiogroups.

Evidence: src/components/songs/SongList.tsx:68, src/components/exercises/ExerciseList.tsx:64, src/components/metronome/MetronomeModeToggle.tsx:12

Fix: add explicit up/down reorder buttons and extract a reusable segmented-control component with roving tabindex and arrow-key behavior.

Recommended Roadmap

First

Make JSON persistence safe

  • Add atomic read-modify-write helper under the server file lock.
  • Clear or remove practice-time.json so reset statistics stays reset.
  • Add focused regression tests for concurrent session appends and reset behavior.
Second

Unify the session runtime

  • Extract shared session lifecycle logic from song and exercise pages.
  • Replace exerciseAsSong with a smaller runtime tempo subject.
  • Make duration and progression rules explicit and validated.
Third

Improve the heavy UI surfaces

  • Fix settings and live-session responsive layout.
  • Collapse advanced template editing by default.
  • Move manual stats logging behind a disclosure or below the dashboard summary.
